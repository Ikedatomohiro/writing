# なぜClaude Codeが選ばれるのか？ 満足度46%で圧倒的1位の理由

「ハーネスエンジニアリング」という言葉を初めて聞いたとき、僕は正直ついていけなかった。

コーディングエージェント、プロンプトエンジニアリング、そしてハーネスエンジニアリング。
AIまわりの概念が次々と生まれて、変化についていくのがやっとだ。

でも、そんな激動の中で1つだけはっきりしたことがある。

**Claude Codeが、開発者満足度46%で圧倒的1位になった。**

2位のCursorが19%、3位のGitHub Copilotが9%。
[Pragmatic Engineer Survey（2026年2月）](https://newsletter.pragmaticengineer.com/p/ai-tooling-2026)の結果は、2位以下に大差をつけるものだった。

2025年5月のリリースからわずか8ヶ月。
2021年以来GitHub Copilotが守ってきた首位を、あっさり奪い取った形になる。

この記事では、なぜClaude Codeがここまで選ばれているのかを掘り下げていく。
スペック比較ではなく「開発体験がどう変わったか」という視点だ。
AIコーディングツールに興味がある人、乗り換えを検討している人に読んでほしい。

---

## いま、AIコーディングツール市場に何が起きているのか

なぜこんなことが起きたのか。
その前に、市場全体で何が起きているか整理しておきたい。

数字がすべてを物語っている。

- **95%**の開発者がAIツールを週に1回以上使っている
- Microsoftの社内コードの**20〜30%**がAI生成（CEOのSatya Nadellaが言及）
- Googleでは新規コードの**30%超**がAI生成（CEOのSundar Pichai、2025年4月のQ1決算発表時）
- AIコーディングツール市場は**128億ドル**規模（2024年の51億ドルから急成長）

もはやAIを使わずにコードを書くほうが珍しい時代になりつつある。

そして、ツールの役割そのものが変わった。
かつてのAIコーディングツールは「タブキーで候補を受け入れる」補完ツールだった。
いまは違う。

Copilot Agent Mode、Cursor Background Agents、Google Antigravityなど、あらゆるプレイヤーが「エージェント」に向かっている。
タスクを丸ごと任せて、AIが自律的にコードを書き、テストし、修正する。

**「補完」から「協働」へ。**
これがいま起きているパラダイムシフトだ。

## Claude Codeが選ばれる理由は「体験」にある

では、全ツールがエージェントに向かう中で、なぜClaude Codeだけが圧倒的な満足度を獲得しているのか。

僕の実感としては、Claude Codeは「ツール」というより「一緒に考えてくれるパートナー」に近い。

実は僕がプログラミングを学び始めたとき、スクールの講師が音信不通になるという最悪の体験をした。
1ヶ月放置されて、ようやくアサインされた次のメンターが非常に丁寧な人だった。
そこからやっと学習が軌道に乗った。

あのとき痛感したのは、**良いメンターがいるかどうかで学習効率は天と地ほど変わる**ということ。
Claude Codeは、24時間いつでも相談できて、コードの意図を理解してくれる。
音信不通にならないメンターが常にそばにいる感覚に近いと思う。

ある開発者はこう語っている。

> 「Claude Codeは速くしてくれたのではない。より良くしてくれた。それがはるかに価値があった」

この言葉が、満足度46%の理由をよく表している。
Redditのブラインドテストでも、Claude Codeは**67%の勝率**を記録した。
作者のBoris ChernyがXに投稿した「箱から出してすぐ使える」という言葉は、**850万閲覧以上・4.9万いいね**を集めている。

### 「コードを書く」から「タスクを任せる」へ

従来のAIコーディングツールは、1行ずつコードを補完するものだった。
Claude Codeはそうじゃない。

マルチファイルのリファクタリング、アーキテクチャの設計、大規模なデバッグ。
こうしたタスクを丸ごと委任できる。

実際、[Anthropic社内ではCode Reviewの導入で、実質的なレビューコメントが付くPRが16%から54%に改善](https://claude.com/blog/code-review)された。
エンジニアのコード生産量は1年で**200%増加**したという。
「コードを書く」から「タスクを任せてレビューする」へ。
開発のやり方自体が変わりつつある。

### 1Mトークンという「記憶力」の差

Claude Codeが他と決定的に違うポイントの1つが、**100万トークンのコンテキストウィンドウ**だ。
2026年3月13日に一般提供（GA）が開始されたこの機能により、大規模なコードベース全体を把握した上で提案してくれる。

実用面で何が変わるかというと、的外れな提案が圧倒的に減る。
モノレポ全体の構造を理解しているから、「このファイルの変更が別のファイルに影響する」といった指摘もしてくれる。

長文脈のベンチマーク（MRCR v2）では、100万トークンの条件でClaude Opus 4.6が**78.3%**を記録。
同条件でGPT-5.4は36%にとどまり、大きな差がついている。

## 16台のAIがCコンパイラを作った話

この「協働」の極限を示す事例がある。

[Anthropicの研究者Nicholas Carliniが実施した実験](https://www.anthropic.com/engineering/building-c-compiler)では、16台のClaude Opus 4.6がそれぞれ独立したDockerコンテナ上で動いた。
共有Gitリポジトリを通じてコラボしながら、**Rust製のCコンパイラを構築**したのだ。

規模感がすごい。

- 約**2,000セッション**を実行
- **10万行**のRustコードを生成
- APIコスト約**2万ドル**（約300万円）
- Linux 6.9をx86/ARM/RISC-Vでコンパイル可能
- GCC torture testで**99%合格**

SSHすら知らなかったところからエンジニアになった僕からすると、AIが16台で協働してコンパイラを作るなんて、ちょっと信じられない光景だ。

これを支えているのが、2026年2月5日にリリースされた**Agent Teams**という機能。
リーダー1台がタスクを分割し、チームメイトが独立して作業しながら直接コミュニケーションを取る仕組みになっている。

ただし、制約もある。
生成されたコードは、GCCの最適化を無効にした場合よりも非効率だった。
Rustのコード品質も専門家レベルには及ばない。
「人間の代わり」ではなく、「人間と協働する相棒」というのが現時点での正確な位置づけだろう。

## それでも知っておくべきリスクと課題

このように可能性は大きいが、課題もまだ多い。
目を背けずに整理しておきたい。

### セキュリティリスクの現実

2026年3月だけで、AI生成コードに起因する**35件のCVE**（共通脆弱性識別子）が公開された。
1月が6件、2月が15件だったことを考えると、急増している。

Veracodeの分析では、**AI生成コードの55%しかセキュアではない**という結果も出ている。

さらに厄介なのが「Slopsquatting」と呼ばれる攻撃手法だ。
AIが存在しないパッケージ名を推奨してしまう現象を悪用する。
756,000サンプル中約20%で発生し、その偽パッケージ名を先取りしてマルウェアを仕込む。
実際に128の偽パッケージが半年で12万回以上ダウンロードされた。

一方で、Anthropicは2026年2月20日に[Claude Code Security](https://www.anthropic.com/news/claude-code-security)をリリースしている。
コードベースのセキュリティ脆弱性をスキャンし、修正パッチを提案する機能だ。
すでに500以上の脆弱性を発見した実績がある。
ツール側も対策を進めてはいるが、AIが書いたコードを無条件に信頼するのは危険だ。

### コストと使用制限の壁

開発者コミュニティで最も多い不満が、**使用制限**だ。

Proプラン（月額20ドル）では、プロンプト数の制限が厳しい。
「12プロンプトで切れるなら日常使いできない」という声もある。
パワーユーザー向けのMaxプランは月額100〜200ドルで、決して安くはない。

ただし、平均的な利用コストは開発者1人あたり約**6ドル/日**。
ある開発者は8ヶ月で100億トークンを使い、API換算で15,000ドル超だった。
Max 100ドル/月なら800ドルで**93%の節約**になったと報告している。

具体的なプラン選びの目安はこうなる。

- **ライトに試したい** → Pro 20ドル/月で始めてみる
- **日常的に使い倒す** → Max 5x 100ドル/月（約88,000トークン/5時間）
- **チーム導入** → Team Premium 100ドル/席/月（最低5席）

### AIに頼りすぎるリスク

もう1つ見逃せないのが、**スキル退化のリスク**だ。

67%の開発者が、AI生成コードのデバッグに人間が書いたコードより多くの時間を費やしている。
AIツールが最大41%多くのバグを導入するという調査結果もある。

特に新人エンジニアが基礎スキルを身につける前にAIに依存すると、トラブル時に自力で対処できなくなる。

だからこそ、**AIが書いたコードを必ず人間がレビューする仕組み**が重要になる。
AIに任せっぱなしにせず、「なぜこのコードが正しいのか」を理解する習慣をつけること。
これは課題であると同時に、エンジニアとしての成長機会でもあると思う。

## 競合ツールとの立ち位置を整理する

Claude Codeの強みと課題を見てきたが、他のツールはどうだろうか。
ここで各ツールの現在地を簡潔に整理しておく。

**Cursor（満足度19%・2位）**
- AI-native IDEとして深いIDE統合が強み
- ARR（年間経常収益）20億ドルを突破（2026年2月時点）
- Background Agentsを搭載し、エージェント機能も充実
- 「IDEから離れたくない」タイプの開発者に人気

**GitHub Copilot（満足度9%・3位）**
- 有料会員470万人、Fortune 100企業の90%が採用
- 月額10ドルからという低価格のエントリーポイント
- VS Code 1.109でClaude/Codex/Copilotを同時実行可能に
- エンタープライズでの導入実績は圧倒的

**OpenAI Codex**
- ChatGPT Plus（月額20ドル）に同梱で手軽
- 発売直後でCursorの60%の利用率を達成する急成長ぶり

現実的には、**70%の開発者が2〜4つのツールを同時利用**している。
「どれか1つ」ではなく、タスクに応じて使い分けるのがいまのスタンダードだ。

エージェント型の複雑なタスクにはClaude Code、IDE統合重視ならCursor、コスト重視ならCopilot。
自分のワークフローに合わせて選ぶのが賢い使い方だろう。

## まとめ

ここまで見てきたように、AIコーディングツールは「補完から協働へ」の転換点にいる。

この記事の要点を整理しておく。

- Claude Codeは8ヶ月で満足度1位（46%）を獲得。理由はスペックではなく**開発体験の変化**にある
- 16台のAIがCコンパイラを構築した事例に象徴される**「協働」**が、今後のスタンダードになりつつある
- セキュリティリスク（AI生成コードの55%しかセキュアでない）やスキル退化のリスクは、理解した上で活用する必要がある
- 競合ツールとの使い分けが現実的。70%の開発者が複数ツールを併用している

今日から始められるアクションとしては、こんな感じだ。

- まずは**Pro 20ドル/月**で試してみる（[Claude公式サイト](https://claude.com/pricing)から登録できる）
- 最初は小さなタスク（テスト生成、リファクタリング）から委任してみる
- AIが書いたコードは**必ずレビューする習慣**をつける

AIコーディングツールの進化は止まらない。
でも、振り回されるのではなく、課題を理解した上でうまく付き合っていく。
そういうスタンスが、いま一番大事なんじゃないかと思う。
